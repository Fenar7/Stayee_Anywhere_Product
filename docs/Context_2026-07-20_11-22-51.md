## 1. Executive Summary & Product Vision
**Anywhere Node** is a next-generation, multi-tenant SaaS platform built for comprehensive hostel and PG (Paying Guest) property management. It provides intelligent infrastructure for real-time dashboards, automated rent collection, digital tenant onboarding, and complex mess/food management.

The platform is designed to serve three distinct user personas, each with their own dedicated portals:
1.  **Main Admin (Organization Owner):** Full visibility across all hostels, financial reporting, and high-level configuration.
2.  **Warden (Hostel Manager):** Daily operational control of a specific hostel, managing occupancy, verifying payments, and handling tenant requests.
3.  **Tenant (Resident):** Self-service portal for paying rent, managing daily food orders, raising maintenance tickets, and tracking their stay lifecycle.

## 2. Core Features & Modules

### A. Multi-Tenant SaaS Architecture
*   **Organizations:** The root entity. All data (Hostels, Users, Leads, Activity) is strictly isolated by `organizationId`.
*   **Role-Based Access Control (RBAC):** Strict routing and API middleware protections ensuring `MAIN_ADMIN`, `WARDEN`, and `TENANT` users only access authorized data.

### B. Property & Inventory Management
*   **Hierarchical Structure:** `Hostel` â†’ `Floor` â†’ `Flat` â†’ `Room` â†’ `Bed`.
*   **Granular Control:** Support for various sharing types (Single, Double, Triple, etc.), bed types (Upper/Lower Berth), and accommodation types (Mens/Womens).
*   **Real-time Occupancy:** Beds are tracked across statuses (`AVAILABLE`, `OCCUPIED`, `IN_MAINTENANCE`, `ON_HOLD`).

### C. Digital Tenant Onboarding
*   **Paperless Flow:** Wardens generate `OnboardingRequest` links sent via SMS/WhatsApp.
*   **Self-Service:** Tenants fill out their profile, upload mandatory KYC documents (Aadhaar, PAN, College/Company ID), and make initial payments.
*   **Auto-generated PDFs:** The system automatically generates Registration Forms and Affidavits for digital records.

### D. Stay & Financial Management
*   **Stay Lifecycle:** Tracks a tenant from `ONBOARDING_PENDING` to `ACTIVE`, `EXTENDED`, `EARLY_EXIT`, or `CHECKED_OUT`.
*   **Billing Engine:** Handles Admission Fees, Security Deposits, Monthly Rent, and Food Charges.
*   **Payments:** Supports Cash, UPI, and Bank Transfers. Tenants upload payment screenshots which Wardens verify.
*   **Receipts & Invoices:** Auto-generates PDF receipts for payments and Refund Invoices for checkouts.

### E. Complex Food & Mess Management
*   **Dynamic Plans:** Tenants can opt for `NOT_INCLUDED`, `BREAKFAST_ONLY`, `BREAKFAST_DINNER`, or `BLD` (Breakfast, Lunch, Dinner).
*   **Billing Modes:** Supports `FLAT_RATE` (fixed monthly), `PREPAID_CONSUMPTION` (wallet-based), and `POSTPAID`.
*   **Daily Ordering:** Tenants can opt-in/opt-out of specific meals daily before a cutoff time (e.g., 8 PM for the next day).
*   **Food Wallets:** Tenants can top-up their food wallets for prepaid consumption, which wardens approve.
*   **Settlement:** Automated generation of `FoodSettlementStatement` at the end of billing cycles.

### F. CRM & Operational Tools
*   **Lead Management:** Track prospective tenants from WhatsApp bots or manual entry.
*   **Ticketing System:** Tenants raise maintenance or cleaning tickets. Tracked via priorities (`LOW` to `CRITICAL`) and statuses (`OPEN` to `RESOLVED`).
*   **Service Requests:** Ad-hoc billing for room changes, fines, or plan upgrades.
*   **Comprehensive Activity Logging:** Every major action (status changes, payments, tickets) is logged in `ActivityLog` for auditing.

## 3. Technology Stack & Architecture

### Backend & Frontend Core
*   **Framework:** Next.js 16 (App Router, Turbopack)
*   **Language:** TypeScript
*   **Database ORM:** Prisma v7
*   **Styling:** Tailwind CSS / UI Components
*   **PDF Generation:** Server-side PDF generation integrated with S3.

### AWS Cloud Infrastructure (Current Deployment)
The application is fully containerized and deployed on AWS, having recently migrated away from a legacy Supabase dependency.

*   **Compute:** Amazon EC2 running a Dockerized Next.js Node container.
*   **Container Registry:** Amazon ECR (`staye-backend:latest`).
*   **Database:** Amazon RDS (PostgreSQL).
*   **Authentication:** 
    *   **AWS Cognito (User Pool):** Serves as the robust identity provider backend.
    *   **NextAuth.js:** Handles session management and JWT creation.
    *   **Custom Login:** A highly customized `/api/auth/login` route bypasses standard OAuth flows to validate plain-text passwords directly against the RDS database for seamless tenant/warden login, wrapping it in a secure NextAuth cookie.
*   **Storage:** Amazon S3 (for Document uploads, PDFs, and Images).
*   **CDN & Routing:** Amazon CloudFront (handles SSL and caching).
*   **Secrets Management:** AWS Systems Manager (SSM) Parameter Store.
*   **CI/CD Pipeline:** GitHub Actions (`deploy.yml`) automates build, push to ECR, and deployment via AWS SSM `RunShellScript`.

## 4. The AWS Migration Journey & Critical Fixes Applied
During the recent transition to AWS, several major architectural roadblocks were resolved:

### A. Prisma v7 Containerization Constraints
*   **The Issue:** Prisma v7 deprecated direct `DATABASE_URL` injections in `schema.prisma`. It requires a dynamic `prisma.config.ts` file which, crucially, needs to execute at *runtime* to resolve environment variables. This caused the Docker container to crash immediately with 504 Gateway Timeouts.
*   **The Fix:** The `Dockerfile` was heavily modified as a multi-stage build. The final runner stage now explicitly copies `prisma.config.ts`, the `prisma` directory, and runs `npm install prisma dotenv` to ensure the Prisma CLI can dynamically resolve the RDS connection string at runtime.

### B. Legacy Authentication De-coupling
*   **The Issue:** The legacy codebase heavily relied on Supabase Auth (`supabase.auth.signInWithPassword`), but the client wrapper (`lib/auth/server.ts`) was merely a dead mock returning `null`. This resulted in guaranteed 500/401 crashes on the login screen. Furthermore, the new AWS RDS database was completely empty.
*   **The Fixes:**
    1.  **Production Seeding:** Created `scripts/seed-prod.mjs` (using raw `pg`). Integrated into the GitHub Actions pipeline, it idempotently creates the root `Organization` and a `MAIN_ADMIN` user *only* if the database is empty.
    2.  **Login Route Rewrite:** Completely rebuilt `app/api/auth/login/route.ts`. It now authenticates users directly against the RDS database using Prisma, validates their password, and manually encodes a NextAuth JWT (`next-auth.session-token`). This successfully bridged the gap between legacy custom auth and the new NextAuth/Cognito architecture.

## 6. Future Recommendations
*   **Password Hashing:** Currently, for legacy compatibility and seeding purposes, `plainTextPassword` is utilized. Transitioning to a secure hashing algorithm (like bcrypt or argon2) should be a priority for production security.
*   **Database Security Groups:** Ensure the RDS security group strictly allows inbound traffic *only* from the EC2 instance's security group.
*   **Log Monitoring:** Continue monitoring AWS CloudWatch logs via the integrated Docker logging driver to catch any edge-case runtime errors.
