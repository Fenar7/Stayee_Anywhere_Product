# Project Context and AWS Migration Status
**Date:** 2026-07-20
**Time:** 11:22:51 (IST)

## 1. Project Overview & Architecture
The project is a Hostel Management Platform (Anywhere Node) designed to handle multi-tenant property management, rent collection, and onboarding.

**Current Tech Stack & Architecture:**
*   **Frontend/Backend Framework:** Next.js 16 (Turbopack)
*   **Database ORM:** Prisma v7
*   **Database:** AWS RDS (PostgreSQL)
*   **Authentication:** 
    *   AWS Cognito (User Pool) for NextAuth integrations.
    *   Custom JWT-based authentication for the main login flow (completely replaced the legacy Supabase Auth).
*   **Containerization:** Docker (Multi-stage build)
*   **Cloud Provider:** AWS
    *   **Container Registry:** Amazon ECR
    *   **Compute/Hosting:** Amazon EC2
    *   **CDN/Routing:** Amazon CloudFront
    *   **Secrets/Config Management:** AWS Systems Manager (SSM) Parameter Store
    *   **Deployment Automation:** AWS SSM `RunShellScript` via GitHub Actions

## 2. The AWS Migration Journey & Challenges Resolved
We successfully migrated the application infrastructure to AWS. The process involved several critical fixes and architectural shifts.

### A. Database and ORM Upgrades
*   **Migrated from Supabase to AWS RDS.**
*   **Prisma v7 Compatibility:** Upgraded to Prisma v7, which introduced breaking changes to how database connections are managed.
    *   **Issue:** Builds failed with `P1012` validation errors because `url = env("DATABASE_URL")` is no longer permitted directly inside `schema.prisma`.
    *   **Fix:** Removed the `url` from `schema.prisma` and implemented `prisma.config.ts` to define the configuration dynamically using `defineConfig`.
*   **Docker Containerization for Prisma:**
    *   **Issue:** The production container was crashing because Prisma v7 requires `prisma.config.ts` and its dependencies (like `dotenv` and a full `node_modules` tree) to be present at *runtime* to resolve the connection URL.
    *   **Fix:** Updated the `Dockerfile` to explicitly copy `prisma.config.ts`, the `prisma` folder, and the `scripts` folder into the final runner stage, and explicitly installed `prisma` and `dotenv`.

### B. Deployment and Infrastructure
*   **GitHub Actions CI/CD (`.github/workflows/deploy.yml`):**
    *   Automated the build, push to ECR, and deployment to EC2 using AWS SSM.
    *   Added `docker system prune` to the deployment script to prevent EC2 disk space exhaustion.
    *   Implemented proper container health checks and CloudWatch log integrations (`--log-driver=awslogs`).
*   **CloudFront 504 Gateway Timeout:**
    *   **Issue:** The application returned 504 errors initially because the Docker container on EC2 was silently crashing due to the missing Prisma runtime configs.
    *   **Fix:** Resolved by fixing the Dockerfile and ensuring the container stays `Up` and binds correctly to port 80.

### C. Authentication Flow Overhaul
*   **Issue:** After fixing the infrastructure, users encountered an "An unexpected error occurred" on the login screen.
    *   The legacy code relied on Supabase Auth (`supabase.auth.signInWithPassword`), but the Supabase client implementation (`lib/auth/server.ts`) was merely a mock that always returned `null`.
    *   Furthermore, the production AWS RDS database was completely empty, meaning no users existed to log in.
*   **Fix 1: Production Database Seeding:**
    *   Created a plain Node.js script (`scripts/seed-prod.mjs`) using `pg` to connect directly to the RDS instance.
    *   The script idempotently creates an initial `Organization` and a `MAIN_ADMIN` user (`admin@nexthome.io`) only if the database is empty.
    *   Integrated this script into the `deploy.yml` pipeline to run immediately after `prisma migrate deploy`.
*   **Fix 2: Complete Login Route Rewrite (`app/api/auth/login/route.ts`):**
    *   Removed the broken Supabase mock dependencies.
    *   Rewrote the route to query the RDS database directly via Prisma.
    *   Implemented plain-text password validation against the database records.
    *   Manually generate a NextAuth-compatible JWT session cookie (`next-auth.session-token`) upon successful validation.
    *   This ensures all downstream route protections (`requireRole()`) that rely on the session cookie function correctly.

## 5. Next Steps / Recommendations
*   **Monitor Deployments:** Keep an eye on the GitHub Actions logs and AWS CloudWatch logs to ensure the container remains stable over time.
*   **Security:** Ensure that the SSM Parameter Store secrets are tightly controlled and rotated as necessary. Ensure the database security groups only allow traffic from the EC2 instance.
*   **Password Hashing:** Currently, the system uses `plainTextPassword` for the initial seed and validation. It is highly recommended to implement standard password hashing (e.g., bcrypt, argon2) for production security as a future enhancement.
