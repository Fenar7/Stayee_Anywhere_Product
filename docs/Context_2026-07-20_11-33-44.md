## 1. Executive Summary & Vision

Anywhere Node is a highly scalable, multi-tenant Software-as-a-Service (SaaS) platform engineered specifically for the management of Hostels, Paying Guest (PG) accommodations, and co-living spaces. The platform aims to digitize and automate the entire lifecycle of property management, tenant relations, financial tracking, and operational logistics.

The system is designed around three primary personas, providing dedicated interfaces and operational capabilities for each:
- **Main Admin (Organization Owner):** Possesses overarching control across multiple properties within an organization. They manage property configurations, view aggregated financial metrics, and handle high-level configurations like food pricing and staff assignment.
- **Warden (Property Manager):** The on-the-ground operational manager for a specific hostel. They manage daily occupancy, process tenant onboarding, verify payments, resolve maintenance tickets, and oversee daily food operations.
- **Tenant (Resident):** The end-user residing in the property. They utilize a mobile-responsive web app to pay rent, upload KYC documents, raise service requests, manage daily meal opt-ins, and view their financial ledger.

## 2. High-Level System Architecture

The application follows a modern monolithic architecture utilizing Next.js 16's App Router for both frontend rendering (SSR/SSG) and backend API route handling.

### 2.1 Multi-Tenancy Strategy
The platform employs a logical multi-tenancy model within a single shared database instance.
- **The `Organization` Entity:** Every piece of operational data (Hostels, Users, Leads, Activity Logs) is strictly bound to an `organizationId`.
- **Data Isolation:** Prisma queries and API middlewares are designed to automatically append `organizationId` filters based on the authenticated user's session, ensuring tenants and wardens cannot access data outside their assigned organization or specific hostel.

### 2.2 Core Modules
1. **Property & Inventory Engine:** Manages the spatial hierarchy of the building (Hostel -> Floor -> Flat -> Room -> Bed) and tracks real-time availability and occupancy states.
2. **Financial & Billing Engine:** A double-entry style ledger tracking charges (Rent, Food, Fines) against payments (Cash, UPI). Handles pro-rated billing, admission fees, and security deposits.
3. **Food & Mess Management:** A highly complex subsystem managing dynamic food plans, prepaid consumption wallets, daily opt-in/opt-out ledgers, and periodic settlement statements.
4. **CRM & Lead Management:** A lightweight CRM to track prospective tenants from initial inquiry to confirmed onboarding.
5. **Operational Ticketing:** An ITIL-style ticketing system for maintenance requests with priority levels, status tracking, and threaded comments.
6. **Digital Document Vault:** Secure storage for KYC documents, payment proofs, and auto-generated legal PDFs (Affidavits, Receipts).

## 4. Database Schema Deep Dive (Data Dictionary)

The database schema is extensive, comprising over 20 models. Below is a detailed breakdown of every entity.

### 4.1 Access Control & Identity Models

#### `Organization`
The root entity of the multi-tenant architecture.
- `id` (UUID): Primary key.
- `name` (String): Display name of the organization.
- `domain` (String): Unique domain for white-labeling.
- `brandColor`, `logoUrl`: UI customization fields.
- **Relations:** 1-to-Many with Hostels, Users, Leads, ActivityLogs, FoodPricings, Tasks.

#### `User`
The central identity model for all personas interacting with the system.
- `id` (UUID): Primary key.
- `supabaseAuthId` (String, Unique): Legacy field name, now used as the internal unique subject identifier for JWTs and Cognito mapping.
- `phone` (String, Unique): Primary contact and login identifier.
- `email` (String, Unique): Secondary contact and login identifier.
- `role` (Enum UserRole): Determines access level (`MAIN_ADMIN`, `WARDEN`, `TENANT`).
- `passwordSetAt` (DateTime): Tracks when the user finalized their account.
- `plainTextPassword` (String): Currently stores the password for the custom login implementation.
- `organizationId` (String): Foreign key to Organization.
- **Relations:** 1-to-1 with Warden, Tenant. 1-to-Many with ActivityLogs, Tasks, Notifications.

#### `Warden`
Extension of the User model for property managers.
- `userId` (String, Unique): Foreign key to User.
- `hostelId` (String, Unique): Foreign key to Hostel (A warden manages exactly one hostel).

#### `Tenant`
Detailed profile information for residents.
- `userId` (String, Unique): Foreign key to User (Nullable during early lead stages).
- `fullName`, `dateOfBirth`, `gender`: Basic demographic data.
- `emergencyContactName`, `emergencyContactNumber`: Critical safety data.
- `occupationType` (Enum): `STUDENT` or `WORKING_PROFESSIONAL`.
- `collegeName`, `companyName`: Demographic tracking.
- `photoUrl`: Path to S3 profile image.
- **Relations:** 1-to-Many with Stays, Documents, Tickets.

### 4.2 Property & Inventory Hierarchy

#### `Hostel`
The primary property entity.
- `id` (UUID): Primary key.
- `name`, `address`: Basic details.
- `accommodationType` (Enum): `MENS` or `WOMENS`.
- `foodOrderCutoffStartHour`, `foodOrderCutoffEndHour`: Configuration for the food module (e.g., locking next day's menu at 8 PM).
- `organizationId`: Link to Organization.
- **Relations:** 1-to-Many with Floors, Stays, Leads, Tickets.

#### `Location`
Geographical grouping for hostels (e.g., "North Campus", "Tech Park").

#### `Floor`
- `hostelId`: Foreign key to Hostel.
- `name`: e.g., "Ground Floor", "1st Floor".
- `sortOrder`: Integer for UI sorting.

#### `Flat`
Optional grouping within a floor (e.g., "Apartment 101").
- `floorId`: Foreign key to Floor.
- `isPrivate`: Boolean indicating if the flat is rented as a single unit.

#### `Room`
The physical room containing beds.
- `flatId` or `floorId`: Must belong to either a Flat or directly to a Floor.
- `roomNumber`: e.g., "101A".
- `sharingType` (Enum): `SINGLE`, `DOUBLE`, `TRIPLE`, `FOUR`, `SIX`, `EIGHT`.
- `isPrivate`: Boolean.

#### `Bed`
The atomic unit of inventory.
- `roomId`: Foreign key to Room.
- `label`: e.g., "A", "B", "Window Side".
- `bedType` (Enum): `UPPER_BERTH`, `LOWER_BERTH`, `SINGLE_COT`.
- `status` (Enum): `AVAILABLE`, `OCCUPIED`, `IN_MAINTENANCE`, `ON_HOLD`, `NOT_IN_USE`.

### 4.3 Lifecycle & Financial Models

#### `OnboardingRequest`
Tracks the digital onboarding flow before a tenant is fully active.
- `phone`: Contact of the prospective tenant.
- `hostelId`, `bedId`, `wardenId`: Target inventory and assigning manager.
- `status` (Enum): `PENDING`, `COMPLETED`, `EXPIRED`.
- `onboardingCurrentStep`: Tracks progress through the UI wizard.

#### `Stay`
The core transactional model linking a Tenant to a Bed for a specific duration.
- `tenantId`, `bedId`, `hostelId`: Core links.
- `status` (Enum): `ONBOARDING_PENDING`, `APPROVED_AWAITING_PAYMENT`, `ACTIVE`, `EXTENDED`, `EARLY_EXIT`, `CHECKED_OUT`, `CANCELLED`.
- `durationType` (Enum): `MONTHLY`, `WEEKLY`, `DAILY`, `CUSTOM`.
- `joiningDate`, `endDate`: Duration tracking.
- `admissionFeePaise`, `monthlyRentPaise`, `securityDepositPaise`: Financial configuration.
- `foodPlan` (Enum): `NOT_INCLUDED`, `BREAKFAST_ONLY`, `BREAKFAST_DINNER`, `BLD`.
- `foodBillingMode` (Enum): `PREPAID_CONSUMPTION`, `POSTPAID`, `FLAT_RATE`.

#### `Payment`
Records of financial transactions.
- `stayId`: Link to Stay.
- `amountPaidPaise`: Amount in lowest currency denomination to prevent float errors.
- `paymentMode` (Enum): `CASH`, `UPI`, `COMPANY_ACCOUNT`, `BANK_TRANSFER`.
- `paymentStatus` (Enum): `PAID`, `PARTIALLY_PAID`, `PENDING`.
- `verifiedByUserId`: Audit trail for Warden verification.
- `screenshotDocumentId`: Link to uploaded proof in S3.
- `receiptNumber`: Auto-incrementing identifier.

#### `RefundInvoice`
Generated upon checkout or cancellation.
- `originalAmountPaise`, `refundAmountPaise`: Calculation data.
- `pdfDocumentId`: Link to the generated S3 document.

### 4.4 Operational Models

#### `Ticket` & `TicketComment`
Maintenance request system.
- `title`, `description`, `category`, `priority`.
- `status` (Enum): `OPEN`, `IN_PROGRESS`, `RESOLVED`, `CLOSED`.
- Comments allow threaded communication between tenant and warden.

#### `ActivityLog`
Immutable audit trail.
- `eventType` (Enum): e.g., `TENANT_ONBOARDED`, `PAYMENT_RECEIVED`, `TICKET_RAISED`.
- `actorId`, `actorName`: Who performed the action.
- `subjectType`, `subjectId`: What was affected.

#### `Lead` & `LeadNote`
CRM entities.
- `phone`, `source` (`WHATSAPP_BOT`, `MANUAL`).
- `status` (`NEW`, `CONTACTED`, `CONVERTED`).

### 4.5 Complex Food Module Entities

#### `FoodPricing`
Defines costs per meal.
- `breakfastPricePaise`, `lunchPricePaise`, `dinnerPricePaise`.
- `effectiveFrom`: Date the pricing takes effect (allows historical accuracy).
- Can be set at the Organization level or overridden at the Hostel level.

#### `FoodOrder`
Daily ledger of tenant meal choices.
- `forDate`: The specific day.
- `breakfast`, `lunch`, `dinner` (Booleans): Opt-in status.
- `lockedAt`: When the cutoff time passed.

#### `FoodBillingCycle`
Periodic grouping of food charges.
- `cycleStart`, `cycleEnd`.
- `status` (`OPEN`, `SETTLING`, `CLOSED`).
- Snapshots pricing at cycle start to prevent mid-cycle cost fluctuations.

#### `FoodWalletTopUp`
Prepaid ledger entries.
- `amountPaise`, `paymentMode`.
- `status` (`PENDING`, `APPROVED`, `REJECTED`).

## 5. Core Business Workflows

### 5.1 The Tenant Onboarding Lifecycle
1. **Lead Generation:** A prospective tenant contacts the hostel. A `Lead` is created manually by the Warden or via an automated WhatsApp bot integration.
2. **Inventory Assignment:** The Warden converts the Lead, selecting a specific `Bed` and configuring the `Stay` parameters (Rent, Deposit, Food Plan). This creates an `OnboardingRequest` and locks the bed (`ON_HOLD`).
3. **Digital Onboarding (Tenant):** The tenant receives a secure link via SMS/WhatsApp. They log into the Tenant Portal.
4. **KYC Upload:** The tenant completes a multi-step form, uploading required `Document` entities (Aadhaar, Profile Photo) directly to AWS S3 via presigned URLs.
5. **Initial Payment:** The system calculates the total payable (Rent + Deposit + Fees). The tenant makes a payment (e.g., via UPI) and uploads the screenshot. A `Payment` record is created in `PENDING` status.
6. **Verification (Warden):** The Warden reviews the KYC documents and the payment screenshot in the Warden Portal.
7. **Activation:** Upon Warden approval, the `Stay` status changes to `ACTIVE`, the `Bed` status changes to `OCCUPIED`, and the system automatically generates a PDF Registration Form stored in S3.

### 5.2 The Daily Food Operations Cycle
1. **Pricing Configuration:** Admin sets global `FoodPricing` for meals.
2. **Tenant Opt-In:** Tenants log into their portal and check/uncheck meals (Breakfast, Lunch, Dinner) for upcoming days via the `FoodOrder` table.
3. **Cutoff Enforcement:** A cron job or database trigger enforces the `foodOrderCutoffStartHour`. For example, at 8:00 PM, all `FoodOrder` records for the *next* day are marked as `lockedAt = NOW()`. Tenants can no longer modify them.
4. **Kitchen Reporting:** Wardens generate aggregate reports of locked `FoodOrder` records for the kitchen staff to prepare the exact quantity of food.
5. **Billing (Prepaid):** If the tenant is on `PREPAID_CONSUMPTION`, the cost of the locked meals is deducted from their `FoodWalletTopUp` ledger for the current `FoodBillingCycle`.

### 5.3 Maintenance Ticketing
1. **Creation:** Tenant raises a `Ticket` via their app (e.g., "AC not cooling").
2. **Triaging:** Warden receives a `Notification` and views the ticket on their dashboard. Warden changes status to `IN_PROGRESS` and assigns a staff member.
3. **Communication:** Warden and Tenant exchange updates via `TicketComment`.
4. **Resolution:** Issue is fixed. Warden marks as `RESOLVED`. Tenant can optionally verify and mark as `CLOSED`.

## 7. Backend API Architecture

The backend consists of Next.js Route Handlers (`app/api/.../route.ts`) acting as RESTful endpoints.

### 7.1 Security Middleware & Access Control
Almost every API route implements a strict authorization pattern using custom utility functions located in `lib/auth/index.ts`:

```typescript
// Example API Route Protection
import { requireRole, requireHostelAccess } from "@/lib/auth";

export async function POST(req: Request) {
  // 1. Authenticate & Authorize Role
  const session = await requireRole(["WARDEN"]);
  
  // 2. Extract context
  const { user } = session;
  const hostelId = user.warden.hostelId;

  // 3. Ensure they own the data they are modifying
  await requireHostelAccess(session, hostelId);
  
  // 4. Proceed with Prisma business logic...
}
```

### 7.2 Major API Domains
- `/api/admin/*`: Endpoints for global configuration (Hostel creation, global food pricing, admin metrics).
- `/api/warden/*`: Scoped endpoints. Operations here strictly filter data based on the Warden's assigned `hostelId`. Includes endpoints for verifying payments (`/stays/[id]/payment`), approving onboarding, and managing local inventory.
- `/api/tenant/*`: Highly scoped endpoints. Data is strictly filtered based on the authenticated Tenant's `id`. Includes endpoints for updating profile, placing food orders, and generating S3 presigned URLs for document upload.
- `/api/pdf/*`: Server-side generation endpoints. Uses libraries like `jspdf` or `pdfmake` to generate binary PDF buffers from HTML/JSON templates, which are then streamed to S3 and returning a URL to the client.

## 8. Comprehensive AWS Migration Post-Mortem

The application recently underwent a major architectural migration from a legacy Supabase backend to a fully native AWS infrastructure. This process was complex and required several critical structural changes to the codebase.

### 8.1 The Initial Goal
To move the database to AWS RDS (PostgreSQL), containerize the Next.js application using Docker, and deploy it to AWS EC2 using GitHub Actions and ECR, utilizing CloudFront as the CDN.

### 8.2 Phase 1: The Prisma v7 Crisis
**The Problem:**
Upon initial containerization, the GitHub Actions build succeeded, but the Docker container immediately crashed upon startup on the EC2 instance, resulting in a **504 Gateway Timeout** from CloudFront.

**The Diagnosis:**
The project was upgraded to Prisma v7. Historically, Prisma required the database connection string to be hardcoded in `schema.prisma` via `url = env("DATABASE_URL")`. Prisma v7 deprecated this, mandating the use of a `prisma.config.ts` file loaded dynamically at runtime via `@prisma/config`.

The initial `Dockerfile` utilized a standard Next.js standalone build. While this copied the compiled application code, it *omitted* the raw `prisma.config.ts` file and the `dotenv` package required to read the `.env` variables injected by the AWS SSM runner. When the container booted, Prisma attempted to initialize, failed to find its configuration, and threw a fatal `P1012` schema validation error, causing the Node process to exit with code 1.

**The Fix:**
The `Dockerfile` was fundamentally rewritten.
1. We modified the multi-stage build to explicitly `COPY --from=builder /app/prisma.config.ts ./` into the final `runner` stage.
2. We added a specific `RUN npm install prisma dotenv` step in the runner stage to ensure the heavy Prisma CLI binaries and environment variable parsers were present in the production image, enabling successful runtime resolution of the RDS connection string.

### 8.3 Phase 2: The Authentication Black Hole
**The Problem:**
With the infrastructure stable, the application loaded the login screen. However, any attempt to log in resulted in a generic "An unexpected error occurred" toast message, with HTTP 500 errors in the network tab.

**The Diagnosis:**
Extensive debugging of the `/api/auth/login` route revealed two overlapping fatal flaws:
1. **The Phantom Database:** The production AWS RDS instance had been successfully migrated (schema applied), but it was completely devoid of data. No Organization or Admin user existed, meaning any database lookup for a user immediately returned `null`.
2. **The Broken Supabase Mock:** The legacy application relied on Supabase Auth. The codebase contained a file `lib/auth/server.ts` that was supposed to interface with Supabase. However, this file contained a hardcoded mock implementation. Specifically, the `signInWithPassword` function was written as `return { data: { session: null }, error: null };`.
   
When the login API route executed, it attempted to look up the non-existent user in the empty RDS database, then passed the credentials to the dead Supabase mock. The mock invariably returned `session: null`, which triggered the application's error handling to throw an unhandled exception, resulting in the 500 Internal Server Error.

**The Fixes:**

**Fix A: Idempotent Production Seeding**
We could not rely on Prisma's standard seeding mechanism (`prisma db seed`) because it requires a full TypeScript environment and development dependencies.
Instead, we created a lightweight, raw Node.js script: `scripts/seed-prod.mjs`.
- Utilizing the standard `pg` driver, this script connects directly to the RDS instance.
- It performs an idempotent check: `SELECT id FROM "Organization" LIMIT 1`.
- If the database is empty, it securely generates UUIDs and inserts the root `Organization` ("Anywhere Node") and the initial `MAIN_ADMIN` user (`admin@nexthome.io` with a `plainTextPassword` of `Password@123`).
- We updated `.github/workflows/deploy.yml` to execute this script via `docker run ... node scripts/seed-prod.mjs` immediately after the `prisma migrate deploy` step, ensuring the database is always initialized before the main application container boots.

**Fix B: Complete Authentication Rewrite**
The legacy dependency on Supabase was completely eradicated from the critical path. We rewrote `app/api/auth/login/route.ts` from scratch to implement a custom authentication flow integrated with NextAuth:
1. The route parses the incoming credentials (Email/Phone and Password).
2. It queries the RDS database directly via Prisma to find the user.
3. It validates the provided password against the `plainTextPassword` field in the database.
4. Upon successful validation, it utilizes the `encode` function from `next-auth/jwt` to manually cryptographically sign a NextAuth-compatible JWT token.
5. It explicitly sets this token as the `next-auth.session-token` cookie on the response, ensuring compatibility with NextAuth's `getServerSession` methodology used throughout the application's middleware for route protection.

## 9. Security & Authentication Architecture

Following the AWS migration, the security posture is structured as follows:

### 9.1 Data Plane Security (Database)
- The AWS RDS instance resides within a private subnet.
- Security Groups restrict inbound traffic on port 5432 exclusively to the Security Group ID associated with the EC2 application instance.

### 9.2 Application Identity (NextAuth + Custom JWT)
- **Session Management:** Stateless JWT cookies (`next-auth.session-token`).
- **Token Claims:** The JWT payload strictly contains the `sub` (mapped to `supabaseAuthId` for legacy structural compatibility), the `id` (database primary key), and the `role`.
- **Role Validation:** Every protected API route utilizes the `requireRole` helper, which parses the JWT, extracts the ID, and re-verifies the user's existence and role directly against the database to prevent privilege escalation via stale tokens.

### 9.3 AWS Cognito Integration
- While the custom login route handles the primary entry point, AWS Cognito User Pools are provisioned (`staye-prod-user-pool`) to handle advanced identity management, potential future OAuth integrations, and secure token signing via NextAuth's `CognitoProvider`.

## 10. Deployment & DevOps Runbook

The application is deployed via a fully automated CI/CD pipeline using GitHub Actions.

### 10.1 Trigger
The workflow (`.github/workflows/deploy.yml`) is triggered on `push` to the `main` branch.

### 10.2 Pipeline Stages
1. **AWS Authentication:** The GitHub runner authenticates with AWS using OpenID Connect (OIDC) or stored Access Keys.
2. **ECR Login:** Authenticates Docker with the Amazon Elastic Container Registry.
3. **Docker Build:** Executes the multi-stage `Dockerfile`.
   - Compiles the Next.js application (Turbopack).
   - Generates the Prisma Client.
   - Bundles the runtime dependencies (`prisma`, `dotenv`).
4. **Docker Push:** Pushes the immutable image tag (usually `latest` and a Git SHA) to ECR.
5. **AWS SSM Deployment:** Uses AWS Systems Manager `send-command` to execute a shell script directly on the target EC2 instance.
   - Pulls the latest image from ECR.
   - Retrieves secrets (DATABASE_URL, NEXTAUTH_SECRET, COGNITO_CLIENT_ID) securely from SSM Parameter Store.
   - Executes `npx prisma migrate deploy` to apply schema changes to RDS.
   - Executes `node scripts/seed-prod.mjs` to initialize data if required.
   - Stops and removes the old container.
   - Starts the new container with the `awslogs` driver, binding port 80 to container port 3000.
   - Runs `docker system prune -f` to clean up old images and prevent EC2 disk exhaustion.
